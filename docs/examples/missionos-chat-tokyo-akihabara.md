# MissionOS Chat: Tokyo Station to Akihabara

This page shows an actual local `missionos chat` run from this public snapshot.
It is an example of the LLM-backed planning path, not proof of flight,
simulator execution, delivery, or physical operation.

## Scenario

The operator asked MissionOS to plan a drone mission from Tokyo Station to
Akihabara Station.

The run was planning-only:

- no operator approval was submitted
- no execution request was prepared
- no simulator was started
- no command ACK was observed
- no route progress or mission completion was claimed

## Command

The sample used the production Gateway backend and local Ollama/Gemma. Local
Gemma can be slow, so the CLI timeout was raised from the default.

```bash
MISSIONOS_GATEWAY_BACKEND=production \
MISSIONOS_LLM_BACKEND=ollama \
MISSIONOS_OLLAMA_MODEL=gemma4:26b \
missionos --timeout 240 chat --autostart \
  "東京駅から秋葉原駅までのドローン任務を計画してください。実行はせず、提案だけしてください。"
```

## What Happened

MissionOS started a Gateway, sent the operator instruction through the chat
conversation route, and returned a bounded mission proposal.

The response stayed on the proposal side of the boundary:

- `route=mission_designer_plan` means the request was treated as mission
  planning.
- `source=...llm_dialogue_router...chief_internal_route_weather_tools` means
  the LLM-backed routing and internal planning tools participated.
- `progress_counted=False` means the conversation did not count as mission
  progress.
- The next prompt suggested approval, but this run exited with `/quit`.

## What This Does Not Prove

This run does not prove:

- operator approval
- dispatch to an executor
- command ACK
- simulator movement
- observed route progress
- mission completion
- physical execution

The text `PX4/Gazebo mission proposal` is part of the proposal generated by the
planning path. It is not evidence that PX4/Gazebo was started.

## Runtime Notes

When this same scenario was first run with the default 45 second CLI timeout,
the production Gateway request timed out before local Gemma returned. Raising
the CLI timeout to 240 seconds allowed the response above to complete.

Hosted Gemini backends are expected to respond faster, but the same authority
boundary still applies: the LLM may propose, the human must approve, and
MissionOS must keep approval, dispatch, ACK, observed progress, and completion
separate.
